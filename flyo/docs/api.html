HTTP API

A small JSON-over-HTTP surface that the built-in UI uses. Everything is cookie-session authenticated; permission tags decide what you can call.

Authentication

Three endpoints govern the session. The session cookie is flyo_sid — HttpOnly, SameSite=Lax.

POST /api/login

JSON body {"user", "pass"}. On success, sets the cookie and returns the resolved identity.

curl -i -c jar -X POST http://localhost:9212/api/login \
  -H 'Content-Type: application/json' \
  -d '{"user":"admin","pass":"secret"}'

# → HTTP/1.1 200 OK
# Set-Cookie: flyo_sid=abc...; Path=/; HttpOnly; SameSite=Lax
# {"authenticated":true,"user":"admin","perms":{"access":true,"list":true,...}}

POST /api/logout

Invalidates every session for the cookie holder's user and clears the cookie.

GET /api/whoami

Returns the current identity. Useful for the UI to decide what to render at load time.

curl -b jar http://localhost:9212/api/whoami
{"authenticated":true,"user":"admin","perms":{...}}

List directory

GET /api/list?path=/sub/dir — requires l.

curl -b jar 'http://localhost:9212/api/list?path=/Photos/2026'

{
  "path": "/Photos/2026",
  "entries": [
    {"name":"Spring trip","is_dir":true,"size":0,"mtime":1741782120},
    {"name":"camera-roll.zip","is_dir":false,"size":268435456,"mtime":1740336000}
  ]
}

Hidden (dot-prefixed) entries appear only if the caller has the S tag. The recycle bin .Trash/ is never listed.

Download with Range support

GET /api/file?path=/clip.mp4 — requires r.

  • Supports Range: bytes=START-END, bytes=START-, and suffix bytes=-N.
  • Returns 206 Partial Content with Content-Range when a range is requested.
  • If the user has the D tag, the response includes Content-Disposition: attachment.
# Stream the first 1 MB
curl -b jar -H 'Range: bytes=0-1048575' \
  'http://localhost:9212/api/file?path=/clip.mp4' -o head.bin

# Resume an interrupted download
curl -b jar -C - 'http://localhost:9212/api/file?path=/big.iso' -o big.iso

Upload

POST /api/upload?path=/dest/name.ext — requires u.

The request body is the file. Flyo writes to a temp sibling and atomically renames once the stream completes — clients never see a half-uploaded file in listings.

# Upload a single file
curl -b jar --data-binary @local.zip \
  'http://localhost:9212/api/upload?path=/backups/local.zip'

# → 200 {"path":"/backups/local.zip","bytes":12345678}

Create folder

POST /api/mkdir?path=/new/nested/folder — requires u. Intermediate folders are created automatically. Returns 409 Conflict if the path already exists.

Rename / move

POST /api/rename?from=/a&to=/b/c — requires m. Works for files and folders. Intermediate folders in the destination path are created if needed.

Delete

POST /api/delete?path=/foo — requires m.

This does not remove files from disk — it moves them into <Root>/.Trash/ with a timestamp prefix:

/share/foo.txt    →    /share/.Trash/.65a9d2c.foo.txt

The recycle bin is intentionally never exposed via the listing API; restore through the file system.

Error codes

StatusWhen
400 Path tried to escape the share root (.., drive prefix, etc.) or query parameters were malformed.
401 Login attempt failed.
403 Authenticated but missing the permission tag for this endpoint.
404 Path does not exist on disk.
409 Destination already exists (mkdir / rename).
416 Range request was unsatisfiable.
500 Underlying I/O error. Server log will have details.
